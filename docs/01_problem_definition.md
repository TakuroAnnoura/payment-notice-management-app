# 課題定義 v1

## 1. 作成背景

キャンペーンに伴うキャッシュバック支払いでは、通常の支払業務とは異なる条件で支払通知書を作成する必要がある。

実務では、営業担当者が作成した支払対象者一覧をもとに、担当者が支払通知書テンプレートへ必要事項を入力していた。
一覧には、会社名、住所、事業者登録番号、支払金額、支払日、内訳などの情報が含まれる。

また、案件によっては税率10％を考慮するパターンと、税金を考慮しないパターンが混在しており、支払条件を正しく判定したうえで支払通知書を作成する必要があった。

## 2. 現場課題

支払通知書を手作業で作成する場合、以下のような課題がある。

### 2.1 転記ミスの発生

会社名、住所、事業者登録番号、支払金額、支払日、内訳などを手入力するため、入力ミスや転記ミスが発生する可能性がある。

特に支払金額や支払日は、誤りがあると支払処理そのものに影響するため、作成後の確認負荷が高くなる。

### 2.2 税区分・支払条件の誤り

キャンペーンによっては、税率10％を考慮するものと、税金を考慮しないものが混在する。
そのため、支払条件を誤って反映すると、支払金額や内訳に誤りが発生する可能性がある。

### 2.3 作成漏れ・確認漏れ

一覧に記載された支払対象者に対して、支払通知書を作成し忘れる可能性がある。
また、作成後にどの通知書が確認済みで、どの通知書が未確認なのかを把握しづらい場合、確認漏れが発生するリスクがある。

### 2.4 重複作成・重複送付のリスク

同一の支払対象に対して支払通知書を重複して作成・送付してしまうと、二重支払につながる可能性がある。

実務上は発行番号や先方との確認によって解消できる場合もあるが、確認・訂正・連絡対応が発生し、業務負荷や信用面のリスクにつながる。

### 2.5 進捗状況の把握がしづらい

Excelや口頭共有で作業状況を管理している場合、作成済み、確認中、修正中、エラー対応中といった状態を担当者間で把握しづらい。

その結果、確認待ちのまま作業が止まったり、同じ作業を複数人が重複して対応したりする可能性がある。

### 2.6 エラー内容・修正履歴が残りにくい

手作業や口頭共有中心の運用では、どのようなミスが発生したのか、誰がいつ修正したのか、再発防止のために何を確認すべきかが記録として残りにくい。

そのため、同じようなミスが繰り返される可能性があり、業務改善につながりにくい。

## 3. 解決方針

本アプリでは、営業担当者から受け取った支払対象者一覧データを取り込み、支払通知書の作成に必要な情報を自動で反映する。

担当者はゼロから手入力するのではなく、作成された支払通知書と元データの整合性を確認する役割に集中できるようにする。

また、作成状況、確認状況、エラー内容、修正履歴をアプリ上で管理することで、作業の進捗と品質を把握しやすくする。

## 4. 初期開発で対象とする範囲

初期開発では、以下の機能を対象とする。

* 支払対象者一覧データの取り込み
* 必須項目のチェック
* 支払通知書データの作成
* 重複データのチェック
* 作成状況・確認状況の管理
* エラー内容の記録
* 作業ログの記録
* 一覧画面での検索・絞り込み
* 簡易的な件数集計

## 5. 初期開発では対象外とする範囲

以下の機能は実務上有効ではあるが、初期開発では対象外とし、今後の拡張機能として扱う。

* 公的な事業者登録番号一覧CSVの自動ダウンロード
* 外部マスタとの事業者登録番号照合
* 毎朝の自動更新処理
* 登録番号の差分チェック
* 実際のメール送付機能
* 会計システムや外部システムとの連携

事業者登録番号については、初期開発では形式チェックや確認ステータスの管理までを対象とし、外部データとの自動照合は将来的な拡張機能とする。

## 6. このアプリで実現したいこと

本アプリでは、支払通知書作成業務における手入力作業を減らし、転記ミス、金額ミス、税区分ミス、作成漏れ、重複作成といったリスクを低減する。

また、作成状況や確認状況を可視化し、エラー内容や作業履歴を記録することで、担当者や管理者が業務の進捗と品質を把握しやすい状態を作る。

単に支払通知書を自動作成するだけでなく、作成後の確認、修正、エラー管理まで含めた業務改善アプリとして設計する。
