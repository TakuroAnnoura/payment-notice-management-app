# 改善バックログ v1

## 1. 目的

本書では、MVP総合テストおよび実装中に判明した改善点を整理する。

改善項目を以下の3段階に分類し、MVPの完成を優先しながら、必要な修正と将来の拡張を管理する。

| 分類        | 内容                             |
| --------- | ------------------------------ |
| 今すぐ対応     | 短時間で修正でき、MVPの使いやすさや分かりやすさに直結する |
| 発表・公開前に対応 | ポートフォリオとしての見栄え、説明力、再現性を高める     |
| 今後の拡張     | 開発量や仕様検討が必要で、MVP完成後に対応する       |

---

## 2. 優先度

| 優先度 | 判断基準                     |
| --- | ------------------------ |
| 高   | 誤操作、見落とし、業務停止につながる可能性がある |
| 中   | 操作性や管理の分かりやすさを改善する       |
| 低   | 現時点では運用可能だが、将来的にあると望ましい  |

---

# 3. 今すぐ対応する改善

## IMP-001 Web画面の金額を3桁区切りで表示する

| 項目   | 内容                    |
| ---- | --------------------- |
| 優先度  | 中                     |
| 対象   | 支払対象一覧、支払対象詳細         |
| 現状   | `110000円`のように表示される    |
| 改善後  | `110,000円`のように表示する    |
| 理由   | 金額を読み取りやすくし、桁の見間違いを防ぐ |
| 対応状況 | 完了                   |

CSVでは、`110,000円`を数値`110000`として正常に保存できている。

今回の改善は保存形式ではなく、Web画面上の表示形式を変更するものである。

---

## IMP-002 ダッシュボードに「取込済み」件数を追加する

| 項目   | 内容                            |
| ---- | ----------------------------- |
| 優先度  | 高                             |
| 対象   | ダッシュボード                       |
| 現状   | 全体件数と各工程の合計が一致しない場合、理由が分かりにくい |
| 改善後  | 通知書未作成の「取込済み」件数を表示する          |
| 理由   | 未着手データの滞留を管理者/SVが把握できるようにする   |
| 対応状況 | 完了                          |

業務ステータスの基本的な内訳は、以下のとおりとする。

```text
全体件数
＝ 取込済み
＋ 確認待ち
＋ 要修正
＋ 確認済み
＋ 完了
＋ 将来追加する対象外
```

処理エラー件数は、この内訳とは別に管理する。

## IMP-003 業務進捗カードとエラーカードを視覚的に分離する

| 項目   | 内容                      |
| ---- | ----------------------- |
| 優先度  | 中                       |
| 対象   | ダッシュボード                 |
| 現状   | エラーが業務ステータスの一段階のように見える  |
| 改善後  | 業務進捗と「要対応エラー」を別の領域に表示する |
| 理由   | 業務の進捗と横断的な問題件数を区別する     |
| 対応状況 | 完了                    |

以下の2軸を明確に分ける。

```text
業務ステータス
取込済み → 確認待ち → 要修正 → 確認済み → 完了
```

```text
処理エラー
未対応 → 対応中 → 解消済み
```

## IMP-004 重複確定データを分かりやすく表示する

| 項目   | 内容                        |
| ---- | ------------------------- |
| 優先度  | 中                         |
| 対象   | 支払対象一覧、支払対象詳細             |
| 現状   | 重複確定後もステータスが「取込済み」のまま残る   |
| 改善後  | 重複確定であることを一覧・詳細で明確に表示する   |
| 理由   | 通知書を作成できない理由を利用者が判断しやすくする |
| 対応状況 | 完了                       |

MVPでは、重複確認結果が「重複確定」の場合に支払通知書作成を停止できている。

今すぐの対応では、一覧や詳細画面の表示を改善する。

「対象外」ステータスの追加は、今後の拡張として別途検討する。

# 7. バックログ更新ルール

改善項目へ着手した際は、対応状況を更新する。

```text
未着手
↓
対応中
↓
完了
```

対応を見送る場合は、理由を記録する。

```text
見送り
理由：MVPの範囲外であり、卒業制作完成後の拡張とする。
```

新しい改善点が見つかった場合は、既存の番号と重複しないIDを付けて追加する。
