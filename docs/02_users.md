# ユーザー像 v1

## 1. 想定ユーザー

本アプリでは、支払通知書作成業務に関わる以下のユーザーを想定する。

* 管理者/SV
* メンバー/作業担当者

初期開発では、主に管理者/SVとメンバー/作業担当者の利用を想定し、閲覧者や上長向けの機能は今後の拡張機能として扱う。

## 2. 管理者/SV

### 2.1 役割

管理者/SVは、支払通知書作成業務全体の進捗、品質、エラー状況を管理するユーザーである。

営業担当者から受け取った支払対象者一覧をもとに、作成状況や確認状況を把握し、作成漏れ、重複作成、エラー対応の遅れが発生していないかを確認する。

### 2.2 主な関心ごと

管理者/SVは、以下の情報を把握したい。

* 支払通知書が何件作成済みか
* 未作成、確認待ち、修正中、完了がそれぞれ何件あるか
* エラーが発生しているデータはどれか
* 重複の可能性があるデータはどれか
* 誰がどの作業を担当しているか
* どの案件の対応が遅れているか
* 修正履歴や作業ログが残っているか

### 2.3 解決したい課題

Excelや口頭共有で進捗を管理している場合、作業状況が担当者ごとに分散し、全体像を把握しづらい。

また、エラーや修正内容が記録として残りにくいため、同じミスが繰り返されたり、対応状況の確認に時間がかかったりする。

本アプリでは、一覧画面やステータス管理、エラー記録、作業ログを通じて、管理者/SVが業務全体を把握しやすい状態を作る。

## 3. メンバー/作業担当者

### 3.1 役割

メンバー/作業担当者は、支払対象者一覧をもとに、支払通知書の作成、確認、修正を行うユーザーである。

本アプリでは、一覧データの取り込み、支払通知書データの確認、エラー内容の修正、確認ステータスの更新を行う。

### 3.2 主な関心ごと

メンバー/作業担当者は、以下の情報を把握したい。

* 自分が対応すべきデータはどれか
* 支払通知書の作成に必要な情報が揃っているか
* 必須項目に不足がないか
* 金額や税区分に問題がないか
* 重複の可能性がないか
* 修正が必要な項目はどれか
* 確認済みにしてよい状態か

### 3.3 解決したい課題

手作業で支払通知書を作成する場合、会社名、住所、事業者登録番号、支払金額、支払日、内訳などを1件ずつ入力する必要があり、転記ミスや確認漏れが発生しやすい。

本アプリでは、一覧データから支払通知書作成に必要な情報を自動反映し、メンバー/作業担当者がゼロから手入力する作業を減らす。

また、必須項目チェックや重複チェックの結果を確認できるようにすることで、作成後の確認作業を行いやすくする。

## 4. 初期開発で扱う権限

初期開発では、以下の2種類の権限を想定する。

| 権限         | できること                                |
| ---------- | ------------------------------------ |
| 管理者/SV     | データ取込、一覧確認、ステータス管理、エラー確認、作業ログ確認、集計確認 |
| メンバー/作業担当者 | 一覧確認、支払通知書作成、確認ステータス更新、エラー修正、作業ログ登録  |

メンバー/作業担当者には、主に「作成担当」と「確認担当」の2つの役割がある。

作成担当は、支払対象者一覧データをアプリに取り込み、支払通知書の自動作成を実行する。

確認担当は、作成された支払通知書データと元の一覧データを照合し、会社名、住所、事業者登録番号、支払金額、支払日、内訳などに誤りがないかを確認する。

ただし、初期開発ではログイン機能や細かい権限制御を必須とせず、画面やデータ設計上で役割を分けることを優先する。


## 5. 今後の拡張ユーザー

今後の拡張として、以下のユーザーを追加することが考えられる。

### 5.1 閲覧者/上長

閲覧者/上長は、作業の詳細操作は行わず、案件全体の進捗やエラー件数、処理完了件数などを確認するユーザーである。

初期開発では対象外とし、ダッシュボードや集計画面の拡張時に検討する。
