# 機能一覧 v1

## 1. 機能一覧の目的

本書では、支払対象者一覧データから支払通知書を自動作成し、作成状況、確認状況、エラー、作業履歴を管理するために必要な機能を整理する。

初期開発では、支払通知書作成業務の基本的な流れを実現することを優先し、外部システム連携や高度な自動化は今後の拡張機能として扱う。

---

## 2. 初期開発で必須とする機能

### 2.1 一覧データ取込機能

| 項目    | 内容                              |
| ----- | ------------------------------- |
| 機能概要  | 営業担当者から受け取った支払対象者一覧データをアプリに取り込む |
| 想定形式  | CSVまたはExcel                     |
| 主な利用者 | 作成担当、管理者/SV                     |
| 主な処理  | ファイル選択、データ読込、取込結果表示             |

取り込む主な項目は以下とする。

* 会社名
* 住所
* 事業者登録番号
* 支払金額
* 支払日
* 内訳
* 発行番号
* キャンペーン区分
* 税区分
* 備考

---

### 2.2 必須項目チェック機能

| 項目    | 内容                         |
| ----- | -------------------------- |
| 機能概要  | 支払通知書作成に必要な項目が入力されているか確認する |
| 主な利用者 | 作成担当、確認担当                  |
| 主な処理  | 未入力項目の検知、エラー表示、対象データの特定    |

以下のような項目をチェック対象とする。

* 会社名が入力されているか
* 支払金額が入力されているか
* 支払日が入力されているか
* 内訳が入力されているか
* 発行番号が入力されているか
* 税区分が設定されているか

---

### 2.3 データ形式チェック機能

| 項目    | 内容                         |
| ----- | -------------------------- |
| 機能概要  | 取り込んだデータが想定した形式になっているか確認する |
| 主な利用者 | 作成担当、確認担当                  |
| 主な処理  | 数値、日付、文字数、形式のチェック          |

チェック例は以下とする。

* 支払金額が数値になっているか
* 支払日が日付形式になっているか
* 事業者登録番号が想定した形式になっているか
* 発行番号が想定した形式になっているか

初期開発では、事業者登録番号の存在確認までは行わず、形式チェックのみを対象とする。

---

### 2.4 重複チェック機能

| 項目    | 内容                         |
| ----- | -------------------------- |
| 機能概要  | 同一の支払通知書が重複して作成される可能性を検知する |
| 主な利用者 | 作成担当、確認担当、管理者/SV           |
| 主な処理  | 発行番号や支払情報を利用した重複判定         |

重複チェックには、以下の項目を利用する。

* 発行番号
* 事業者登録番号
* 会社名
* 支払金額
* 支払日
* キャンペーン区分

発行番号が一致する場合は、原則として重複エラーとして扱う。

発行番号が異なる場合でも、会社名、支払金額、支払日などが一致する場合は、重複の可能性があるデータとして警告を表示する。

---

### 2.5 支払通知書作成機能

| 項目    | 内容                         |
| ----- | -------------------------- |
| 機能概要  | 取り込んだ一覧データをもとに支払通知書を自動作成する |
| 主な利用者 | 作成担当                       |
| 主な処理  | テンプレートへのデータ反映、支払通知書データの生成  |

アプリは、以下の情報を支払通知書に反映する。

* 会社名
* 住所
* 事業者登録番号
* 支払金額
* 支払日
* 内訳
* 発行番号
* 税区分

支払通知書の出力形式は、設計段階でExcelまたはPDFなどから決定する。

---

### 2.6 作成結果確認機能

| 項目    | 内容                             |
| ----- | ------------------------------ |
| 機能概要  | 作成された支払通知書と元の一覧データを比較し、内容を確認する |
| 主な利用者 | 確認担当                           |
| 主な処理  | 元データと作成結果の表示、確認結果の登録           |

確認担当は、以下の内容を確認する。

* 会社名
* 住所
* 事業者登録番号
* 支払金額
* 支払日
* 内訳
* 発行番号
* 税区分

確認後、以下のいずれかの状態に更新する。

* 確認済み
* 要修正
* エラー

---

### 2.7 ステータス管理機能

| 項目    | 内容               |
| ----- | ---------------- |
| 機能概要  | 各支払通知書の作業状況を管理する |
| 主な利用者 | 作成担当、確認担当、管理者/SV |
| 主な処理  | ステータス表示、ステータス更新  |

初期開発で使用する主なステータスは以下とする。

* 未処理
* 取込済み
* 作成済み
* 確認待ち
* 確認済み
* 要修正
* エラー
* 完了

ステータスの詳細な遷移順は、画面設計および業務フロー設計の段階で整理する。

---

### 2.8 エラー管理機能

| 項目    | 内容                          |
| ----- | --------------------------- |
| 機能概要  | データ取込や支払通知書作成時に発生したエラーを記録する |
| 主な利用者 | 作成担当、確認担当、管理者/SV            |
| 主な処理  | エラー登録、エラー内容表示、対応状況更新        |

記録する主な内容は以下とする。

* 対象データ
* エラー種別
* エラー内容
* 発生日時
* 対応状況
* 対応内容
* 対応者
* 解消日時

エラーの対応状況は、以下を想定する。

* 未対応
* 対応中
* 解消済み

## 3. 余裕があれば追加する機能

初期開発の必須機能が完成した後、開発期間に余裕がある場合は以下を追加する。

### 3.1 一括処理機能

* 複数の支払通知書を一括作成する
* 複数データのステータスを一括更新する
* 複数の作成済みファイルをまとめて出力する

### 3.2 ダッシュボード機能

* ステータス別件数をカード形式で表示する
* エラー件数を表示する
* 重複警告件数を表示する
* 担当者別の処理件数を表示する

### 3.3 担当者設定機能

* 作成担当者を設定する
* 確認担当者を設定する
* 未担当のデータを抽出する

### 3.4 コメント・申し送り機能

* 確認担当から作成担当への修正依頼を登録する
* 修正理由や確認事項を記録する
* 担当者間の申し送りを残す

---

## 4. 将来の拡張機能

以下は、初期開発では対象外とする。

### 4.1 事業者登録番号の外部照合

* 公的な登録番号一覧CSVを取り込む
* 事業者登録番号が存在するか確認する
* 登録情報との不一致を検知する

### 4.2 外部データの自動更新

* 公的CSVを定期的にダウンロードする
* 前回データとの差分を確認する
* 追加・変更された登録情報を更新する

### 4.3 メール送付機能

* 作成した支払通知書をメールに添付する
* 送付先を管理する
* 送付日時や送付結果を記録する

### 4.4 外部システム連携

* 会計システムとの連携
* 営業管理システムとの連携
* ストレージサービスとの連携

### 4.5 詳細なログイン・権限管理

* ユーザーごとのログイン
* 管理者、作成担当、確認担当の権限制御
* 操作可能な画面や機能の制限

### 4.6 作成結果の自動整合性チェック

* 元の支払対象者一覧と、作成された支払通知書の内容を自動で照合する
* 会社名、住所、事業者登録番号、支払金額、支払日、内訳、発行番号、税区分の一致を確認する
* 不一致があるデータをエラーまたは確認対象として抽出する
* 正常と判定されたデータの中から、一定件数または一定割合をランダムに抽出する
* 確認担当者は、不一致データとランダム抽出されたデータを中心に確認する
* 自動チェック結果、確認者、確認日時、確認結果を作業ログに記録する

初期開発では人による全件確認を前提とし、自動整合性チェックおよびランダム抽出による確認は今後の拡張機能として扱う。

## 5. 初期開発の優先順位

初期開発では、以下の順に実装する。

1. 一覧データ取込
2. 必須項目・形式チェック
3. 重複チェック
4. 一覧表示
5. 支払通知書作成
6. 作成結果確認
7. ステータス管理
8. エラー管理
9. 作業ログ
10. 検索・絞り込み
11. 詳細表示・修正
12. 簡易集計

まずは、一覧データを取り込み、問題のあるデータを検知し、支払通知書を作成できるところまでを最優先とする。

その後、確認、修正、エラー、作業ログ、集計の機能を追加し、業務全体を管理できる状態を目指す。
